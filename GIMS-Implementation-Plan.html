GIMS Implementation Plan

Gradesmatch Integrated Messaging System — a phased implementation roadmap for building unified student communications with delivery cascade, event-driven messaging, and an AI chatbot.
Version: 2.0 Date: 2026-04-09 Status: Approved Spec: GIMS IDEAL Specification

1. Executive Summary

GIMS replaces the current fragmented communication approach — where WhatsApp, email, SMS, and in-app notifications operate independently — with a single unified messaging platform.

What GIMS Delivers
  • A 3-tier delivery cascade (App → WhatsApp → SMS) with per-message tracking
  • Structured response collection (yes/no, select, rating, text, file upload, etc.)
  • Event-driven auto-messaging triggered by business actions (no code changes for new messages)
  • A single unified AI chatbot replacing 4 fragmented bots
  • Full analytics dashboard measuring app engagement vs WhatsApp dependency
  • Campaign authoring with multi-channel content and audience targeting
What Already Exists
  • gims-api repo with Azure Functions scaffolding, MongoDB client, BridgeService base class, all collection constants, blob storage, email sender
  • WhatsApp sending via WhatsappSend.SendWhatsapp() in bridge-universe
  • SMS sending via SMS.BulkSend() in bridge-universe
  • Email sending via Mail.SendEmail() in bridge-universe
  • Campaign polling pattern via CampaignListener.cs in bridge-universe
  • Meta webhook handler in azf-gradesmatch-identity/metaWebhook.cs
  • Notification prototypes in gradesmatchid and bridge-one-3

4. Guiding Principles

PRINCIPLE 01
No Data Migration
Legacy campaign data stays in gradesmatch_social.whatsapp_campaign. GIMS writes to its own collections. Both systems coexist until legacy is retired.
PRINCIPLE 02
No Direct External API Calls
All external channel sending goes through bridge-universe's existing WhatsappSend, SMS.BulkSend(), and Mail.SendEmail(). GIMS calls these via HTTP endpoints.
PRINCIPLE 03
All Apps from Phase 1
Not just gradesmatch_2026 — bridge-one-3, gradesmatchid, and futuristic all get the comms page simultaneously.
PRINCIPLE 04
Independently Deployable Phases
Each phase is independently deployable and valuable. No phase depends on a future phase to deliver value.
PRINCIPLE 05
Follow Existing Patterns
BridgeService base class, BridgeServicePolicy attributes, MongoCollections constants, Service Generator API pattern, SuperTableGeneric for admin tables.

6. Phase 2 — Delivery Cascade & Channel Integration

02
Delivery Cascade & Channel Integration
Duration: 3–4 weeks Repos: gims-api, bridge-universe, azf-gradesmatch-identity, bridge-console
Goal

Messages cascade App → WhatsApp → SMS with per-message tracking.

Delivery Cascade Tiers

Tier 1 — In-App
App Notification
  • Message delivered to student inbox in all apps
  • Free, rich content, structured responses
If not read within timeout → escalate to Tier 2
↓
Tier 2 — WhatsApp
WhatsApp Message
  • Sent via bridge-universe WhatsappSend.SendWhatsapp()
  • Familiar channel, high open rate, 24h session window
If not delivered/read within timeout → escalate to Tier 3
↓
Tier 3 — SMS
SMS Fallback
  • Sent via bridge-universe SMS.BulkSend()
  • Last resort, highest reach, cost per message
Cascade complete after delivery or 48h timeout

Timeouts by Priority

PriorityTier 1 → 2Tier 2 → 3
Normal3 hours24 hours
Important2 hours12 hours
Urgent30 minutes6 hours

Step 2.1 — Cascade Processor gims-api

Console App following the CampaignListener.cs polling pattern.

Files to Create
FileDetails
CascadeProcessor/Program.csEntry point (new project)
CascadeProcessor/CascadeWorker.csProcessing loop
CascadeProcessor/cascadeprocessor.config.jsonConfiguration
Processing Loop 
WHILE true:
  Query messages WHERE cascadeComplete = false
  FOR each:
    Tier 1 read? → DONE (finalChannel = "app")
    Tier 1 timeout? → Send WA via bridge-universe endpoint → Tier 2
    Tier 2 read? → DONE
    Tier 2 timeout + not delivered? → Send SMS via bridge-universe → Tier 3
    Tier 3 delivered or 48h? → DONE
  Adaptive sleep: 1 min active, 5 min idle
